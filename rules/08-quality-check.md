# 08 质量检查规则

## 目标

交付前检查 `design.md` 是否真的能给 HTML 原型生成 Skill 使用。

## 必检项

### 来源完整性

- 是否有 Figma 来源表；
- 每个输入链接是否进入 manifest；
- 每个正式组件是否有 nodeId；
- 每个页面 example 是否有 nodeId。

### 组件清单

- 组件清单是否包含组件名、类别、Figma 节点、契约位置、截图、状态；
- 清单中的组件是否有对应契约；
- 契约中的组件是否回填到清单；
- 页面模板是否误写进组件清单。

### 组件契约

每个组件契约是否包含：

- class；
- 尺寸；
- 子结构；
- 状态变体；
- 使用场景；
- 生成规则；
- 禁止事项；
- 截图引用。

### 页面 Example

每个页面 example 是否包含：

- 页面用途；
- ASCII 骨架；
- 总览截图；
- 关键区域说明；
- 依赖组件；
- 布局规则；
- 交互状态；
- 禁止事项。

### 截图

- 所有截图路径是否存在；
- 是否存在短效外链；
- 是否存在过宽过扁截图；
- 重要页面是否补局部截图；
- 重要组件是否有清晰状态截图；
- 截图数量是否超过主文档引用上限。

### 文件规模

- 组件契约总数是否超过 20；
- `design.md` 是否超过 30000 中文字符；
- 单组件契约是否超过 1200 中文字符；
- 是否需要拆到 `references/contracts/`。

### 写作质量

- 是否存在空泛审美描述；
- 是否缺少执行词；
- 是否没有 class；
- 是否只有截图没有规则；
- 是否有 TODO、待补、待确认未汇总；
- 是否中英文命名混乱。

## 检查报告格式

输出 `design-md-check.md`，包含：

```text
检查时间
检查范围
通过项
风险项
缺失项
截图问题
建议补充的 Figma 节点
建议补充的页面 example
建议拆分的组件契约
最终结论
```

## 结论等级

```text
可用于原型生成
可用但建议补充
需修复后再使用
无法使用
```

## 交付回复

最终回复用户时应简短说明：

- 新增或更新了哪些组件；
- 新增或更新了哪些页面 example；
- 截图是否已补清晰；
- 是否存在待确认项；
- 文件路径。
